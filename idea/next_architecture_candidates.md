## 1. 現在「おぼろげに見えている構造」の整理

Bigram HMM（k=7/k=8）の分析で浮かび上がった構造は3種類あり、
それぞれの**本質的な性質**が異なる。

| 構造 | 本質 | 観測された証拠 |
|------|------|--------------|
| ① 役割曖昧性 | **局所文脈**の違い | `o` が基末尾か基先頭かで前後の状態が違う。実証遷移 S4→S4 = 0.51 |
| ② 語頭距離勾配 | **絶対位置**の関数 | S4 出現率: 2基76%→3基49%→4基33% と単調減少 |
| ③ 複合語の基境界 | **セグメント構造** | V8 × HMM が独立に同一構造を検出（Fisher B-end vs S-mid, odds=29.9） |

この3種類の構造を同時に捉えるには、Bigram HMM に本質的な限界がある。

## 2. アーキテクチャ比較

### 2.1 Trigram HMM（次数を上げる）

**仕組み**：遷移確率を P(s_t | s_{t-1}) から P(s_t | s_{t-1}, s_{t-2}) に拡張する。

```
Bigram:  P(s_t | s_{t-1})
Trigram: P(s_t | s_{t-1}, s_{t-2})
→ 遷移行列が k×k から k²×k に拡張（パラメータ数 k² 倍）
```

**効く部分：役割曖昧性（①）**

`o` が基末尾か基先頭かは、2ステップ前の状態が違う：

```
基末尾の o: ... [基内部状態 S2/S5] → S4(o)   ← s_{t-2}=内部状態
基先頭の o: ... [直前の基の末尾 S4] → S4(o)   ← s_{t-2}=S4
```

Trigram は この遷移文脈の違いを直接学習できる。
うまくいけば「S4_基末尾」と「S4_基先頭」に相当する2状態への分離が起きる可能性がある。

**効かない部分：語頭距離勾配（②）**

「語頭から何文字目か」という絶対位置を保持するには、BOS からの全ステップ分の
状態履歴が必要になる。Trigram は2ステップしか遡れないため、語長が長くなるほど
位置情報が希薄化し、勾配構造を直接表現できない。

**実用上の問題点**

- パラメータ数が k² 倍になるため、8,060語 という限られたデータでは過学習リスクが高い
- Phantom State がさらに増加する可能性（k=7 で1個、k=8 で1個がすでに発生）
- それでも**実装コストが最低**のため、まず試みる価値はある

### 2.2 位置埋め込みを持つアーキテクチャ

#### CRF（条件付き確率場）

**仕組み**：遷移スコアと放出スコアに加え、**任意の素性関数**を定義できる識別モデル。
位置情報を素性として直接利用できる。

```python
# 素性関数の例（概念コード）
f(state, char, position) = [
    (state == S4) * (position / word_length),   # 語頭からの相対距離
    (state == S4) * (position == 0),             # 語頭フラグ
    (state == S4) * (position == word_length-1), # 語末フラグ
    ...
]
```

これにより「語頭に近いほど S4 が活性化する」という勾配を
**1パラメータで表現**できる（Trigram が多数の状態で近似しようとする構造を直接符号化）。

**利点**：
- 語頭距離勾配（②）を直接モデル化
- 役割曖昧性（①）も位置情報で自然に解消
- HMM より解釈可能（素性の重みが読める）

**問題点**：
- 識別モデルのため本来はラベルが必要
- 教師なしで使うには EM との組み合わせが必要（実装コスト中〜高）
- 素性設計の判断が結果に強く影響するため、設計者の事前仮定が入り込む

#### Transformer

**利点**：
- 自己注意により全位置ペアを直接比較
- 位置エンコーディングで絶対位置を陽に表現

**問題点**：
- ヴォイニッチ語は語長6〜8文字と極めて短く、自己注意の優位性が発揮されにくい
- 教師なし構造発見には VAE・PCFG 的アプローチが必要で実装難度が高い
- Bigram HMM との比較可能性・解釈可能性が大きく低下する

---

### 2.3 各アーキテクチャの構造捕捉能力まとめ

| 構造 | Bigram HMM | Trigram HMM | CRF + position | Transformer |
|------|-----------|------------|---------------|-------------|
| ① 役割曖昧性 | △（S4 に混在） | ○（文脈で分離可） | ○（位置素性で分離） | ○ |
| ② 語頭距離勾配 | ✗（直接不可） | △（間接的にのみ） | ◎（直接モデル化） | ◎ |
| ③ 基境界のセグメント構造 | △（間接的） | △ | ○ | ○ |
| 実装コスト | 済 | 低 | 中 | 高 |
| 解釈可能性 | ◎ | ◎ | ○ | △ |
| データ量への要求 | 低 | 中（k²倍） | 中 | 高 |

## 3. Semi-Markov HMM：中間的な最有力候補

### 3.1 概要

通常の HMM はすべての状態遷移が「1文字ごと」に起きるが、
**Semi-Markov HMM（HSMM）** は状態が**可変長のセグメント**を占有できる。

```
通常の HMM:
  状態: S S S S S S S S
  文字:  o k e e d y s h

Semi-Markov HMM:
  セグメント: [S4: okeedy] [S2: sh]
  各セグメントは可変長で、状態が1文字ではなく「基」単位を占有する
```

これはヴォイニッチ語の複合語構造と**直接対応する**：
- 各基（ベース）= 1つの HSMM セグメント
- セグメントの遷移 = 基から基への移動 = 複合語境界

### 3.2 通常 HMM との違い

| 特性 | 通常の HMM | Semi-Markov HMM |
|------|----------|----------------|
| 状態の持続 | 1文字ずつ遷移 | d 文字間持続（d は確率的） |
| 放出分布 | P(char \| state) | P(char_sequence \| state, duration) |
| セグメント長 | 陰的（Geometric 分布） | 陽的（任意の離散分布） |
| 適切なモデル化 | 文字レベルの局所統計 | 語・基レベルの構造 |

通常の HMM では状態持続時間が幾何分布（Geometric）に固定される。
HSMM では持続時間分布 P(d | state) を自由に学習できるため、
「V8 の各基は何文字か」という分布を直接学習する。

### 3.3 HSMM がこの研究に有効な理由

**① 役割曖昧性の解消**

HSMM では同じ文字でも「セグメント先頭」か「セグメント末尾」かが
セグメント内の位置として陽に定義される。
`o` がセグメントの先頭文字か末尾文字かを**モデルが区別して学習**できる。

```
HSMM のセグメント内放出:
  P(char | state, position_in_segment)
  → "セグメント先頭の o" と "セグメント末尾の o" が異なる確率を持てる
```

**② 語頭距離勾配の自然な表現**

セグメント番号（第1基、第2基、第3基）が遷移構造として陽に記録される。
「第1基の末尾」が「第2基の末尾」より S4 が高いという勾配は、
セグメント順序番号の関数として学習できる。

**③ 基境界の直接的な確率的定義**

HSMM の Viterbi デコードはセグメント分割を直接出力する：

```
入力:    o k e e d y s h
出力: [okeedy | sh]  （確率的な最適分割）
```

これは V8 の `find_v8_splits_first()` と**同じタスクを確率的に実行**することを意味する。
両者の一致度を測定することで、複合語仮説の最も直接的な検証が可能になる。

### 3.4 HSMM の学習アルゴリズム概要

Baum-Welch（EM アルゴリズム）を HSMM 用に拡張する。
計算量は通常 HMM の O(T) から O(T²) に増加するが、ヴォイニッチ語の語長（〜8文字）
では実用的な範囲内。

```python
# 概念的な HSMM の前向きアルゴリズム
for t in range(T):            # 現在位置
    for s in range(K):        # 状態
        for d in range(1, D): # セグメント長（最大 D）
            if t - d >= 0:
                alpha[t][s] += sum(
                    alpha[t-d][s_prev] * trans[s_prev][s]
                    for s_prev in range(K)
                ) * duration_prob[s][d] * emission_prob[s][x[t-d:t]]
```

### 3.5 HSMM の実装における注意点

**放出モデルの設計選択**：
- 単純版：セグメント内の各文字を独立に放出（既存 HMM と同等の放出分布）
- 拡張版：セグメント内位置依存の放出（セグメント先頭/末尾で異なる確率）

後者が①の役割曖昧性解消に直接効く。ただし実装複雑度が上がる。

**セグメント長の上限 D**：
V8 の各基の文字数分布から設定できる（おおよそ 2〜8 文字程度）。

## 4. 推奨ロードマップ

段階的にアーキテクチャの複雑度を上げていく戦略：

```
Step 1: Trigram HMM（実装コスト：低）
  目的: 役割曖昧性（①）が局所文脈で解消されるか検証
  判定: S4 が「基末尾」と「基先頭」の2状態に分離するか
  ↓
  [S4 が分離する場合]         [S4 が分離しない場合]
  → 局所文脈で十分             → 位置情報が本質的
  → Trigram HMM で継続        → Step 2 へ
                               ↓
Step 2: Semi-Markov HMM（実装コスト：中）
  目的: 基（ベース）単位のセグメント構造を直接モデル化
  判定: HSMM の Viterbi 分割と V8 の find_v8_splits_first() が一致するか
  → 複合語仮説の最も直接的な検証が可能
  ↓
  [一致する場合]              [一致しない場合]
  → 複合語構造が確定           → V8 文法の見直し or より高次モデルへ
  → 定量的な複合語確率が得られる

Step 3: CRF with position features（実装コスト：中〜高）
  目的: 語頭距離勾配（②）を直接モデル化、複合語境界の識別
  位置素性: pos/word_length, is_word_initial, is_segment_initial 等
```

## 5. 結論

| 候補 | 主な解決対象 | 推奨優先度 |
|------|-----------|----------|
| Trigram HMM | 役割曖昧性（局所文脈） | **1番目**（コスト低、既存比較可能） |
| **Semi-Markov HMM** | **セグメント構造（全3構造）** | **2番目（最有力の理論的候補）** |
| CRF + position | 語頭距離勾配（絶対位置） | 3番目（Trigram 結果次第） |
| Transformer | 全般 | 最終手段（解釈可能性コスト大） |

**Semi-Markov HMM が最有力**な理由：
ヴォイニッチ語の「複合語＝複数の基の連結」という構造仮説を
**モデルの設計思想そのものとして組み込んでいる**。
Bigram HMM が「文字ごとの状態遷移」として間接的に捉えていた複合語構造を、
HSMM は「セグメントごとの状態遷移」として**直接的に確率的定式化**できる。
検証したい仮説とモデルの帰納バイアスが最も整合している。
